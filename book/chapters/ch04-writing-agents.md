# Chapter 4: 에이전트 정의 작성하기

## 이 챕터에서 배울 것

- 에이전트 파일의 전체 구조: frontmatter와 본문이 각각 어떤 역할을 하는지
- frontmatter의 세 필드(`name`, `description`, `model`)를 올바르게 작성하는 법
- 본문의 다섯 가지 핵심 섹션: 역할 선언, 핵심 역할, 작업 원칙, 출력 형식, 협업을 설계하는 법
- `description`에 트리거 키워드를 효과적으로 배치하는 전략
- `sonnet`과 `opus` 모델을 작업 특성에 맞게 선택하는 기준
- 좋은 에이전트 정의와 나쁜 에이전트 정의의 차이

---

## 본문

### 에이전트 파일은 한 장의 이력서다

Chapter 1에서 에이전트가 "누가 하는가"를 정의한다고 배웠다. Chapter 2에서는 여러 에이전트가 팀을 이루는 구조를 살펴보았다. 이제 직접 에이전트 파일을 작성할 차례다.

에이전트 파일은 한 장의 이력서와 같다. 이력서에는 이름, 직무 요약, 핵심 역량, 경력 사항이 담긴다. 에이전트 파일에는 식별 정보(frontmatter), 역할 선언, 작업 원칙, 출력 형식, 협업 관계가 담긴다. 이력서를 잘 쓰면 채용 담당자가 적합한 포지션에 배치하듯, 에이전트 파일을 잘 쓰면 Claude Code가 적합한 상황에서 해당 에이전트를 활성화한다.

에이전트 파일의 전체 구조를 먼저 조감하자.

```markdown
---
name: {에이전트-이름}
description: "{역할 설명}. '{트리거 키워드1}', '{트리거 키워드2}' 요청 시 트리거."
model: {sonnet 또는 opus}
---

# {에이전트 타이틀} — {한 줄 역할 설명}

당신은 {역할}을 담당하는 전문가입니다.

## 핵심 역할
1. ...
2. ...

## 작업 원칙
- ...
- ...

## 출력 형식
(필요한 경우)

## 협업
- 입력: ...
- 출력: ...
```

크게 두 영역이다. 파일 상단의 **frontmatter**(YAML 메타데이터)와 그 아래의 **본문**(마크다운 지시문). frontmatter는 Claude Code가 에이전트를 식별하고 라우팅하는 데 사용하고, 본문은 에이전트가 실제로 일할 때 참조하는 행동 지침이다.

이 두 영역을 하나씩 뜯어보자.

---

### frontmatter 작성법

frontmatter는 `---`로 감싸진 YAML 블록이다. 세 개의 필드가 있다.

```yaml
---
name: code-reviewer
description: "코드 리뷰를 수행하는 에이전트. '코드 리뷰', 'code review', 'PR 리뷰' 요청 시 트리거."
model: sonnet
---
```

#### name: 고유 식별자

`name`은 에이전트의 ID다. 다른 에이전트나 스킬에서 이 이름으로 참조한다.

작성 규칙:
- **kebab-case** 사용: `code-reviewer`, `book-planner`, `data-analyst`
- 역할을 직관적으로 드러내는 이름: `reviewer`보다 `code-reviewer`가 명확하다
- 프로젝트 내에서 유일해야 한다

나쁜 예와 좋은 예를 비교하자.

| 나쁜 name | 이유 | 좋은 name |
|-----------|------|-----------|
| `agent1` | 역할이 드러나지 않음 | `code-reviewer` |
| `codeReviewer` | camelCase는 파일명과 불일치 | `code-reviewer` |
| `reviewer` | 범위가 모호함 (무엇의 리뷰어?) | `code-reviewer` |
| `code_reviewer` | snake_case는 관례와 다름 | `code-reviewer` |

`name`은 파일명과 일치시키는 것이 관례다. `name: code-reviewer`이면 파일명은 `code-reviewer.md`다.

#### description: 역할 설명 + 트리거 키워드

`description`은 두 가지 역할을 동시에 수행한다.

1. **역할 설명**: 이 에이전트가 무엇을 하는지 한 문장으로 설명
2. **트리거 키워드**: 사용자가 어떤 표현을 사용할 때 이 에이전트가 활성화되는지 명시

Claude Code는 사용자의 요청을 각 에이전트의 `description`과 매칭한다. 사용자가 "코드 리뷰해줘"라고 입력하면, description에 "코드 리뷰"가 포함된 에이전트가 선택된다. 따라서 description은 단순한 설명이 아니라 **검색 인덱스**처럼 기능한다.

트리거 키워드를 효과적으로 넣는 전략은 뒤의 심화 섹션에서 자세히 다룬다. 여기서는 기본 구조만 확인하자.

```yaml
# 기본 패턴
description: "{역할 설명}. '{키워드1}', '{키워드2}', '{키워드3}' 요청 시 트리거."
```

#### model: AI 모델 선택

`model` 필드는 이 에이전트가 사용할 AI 모델을 지정한다. 현재 주요 선택지는 두 가지다.

- **`sonnet`**: 빠르고 비용 효율적. 구조화된 분석, 검증, 패턴 기반 생성에 적합.
- **`opus`**: 느리지만 강력. 복잡한 추론, 긴 텍스트 창작, 미묘한 판단에 적합.

모델 선택은 에이전트의 **작업 특성**에 따라 결정한다. 이에 대한 상세 가이드는 심화 섹션에서 다룬다.

---

### 본문 섹션별 작성법

frontmatter 아래의 본문은 에이전트의 행동 지침이다. 다섯 가지 핵심 섹션으로 구성된다.

#### 섹션 1: 역할 선언

본문은 제목과 역할 선언 문장으로 시작한다.

```markdown
# Code Reviewer — 코드 리뷰 전문가

당신은 소프트웨어 프로젝트의 코드 품질을 검증하는 전문 리뷰어입니다.
```

**제목 형식**: `# {영문 타이틀} — {한 줄 한국어 설명}`

제목 바로 아래의 "당신은 ~입니다" 문장이 중요하다. 이 문장이 에이전트의 **페르소나(persona)**를 설정한다. 페르소나란 AI가 응답할 때 취하는 관점과 어조를 결정하는 역할 정의다. "전문 리뷰어"라고 선언하면, 이후의 모든 출력에서 리뷰어의 관점을 유지한다.

역할 선언 시 주의할 점:

- **구체적으로 쓴다**: "전문가"보다 "코드 품질을 검증하는 전문 리뷰어"가 낫다
- **한 문장으로 한정한다**: 길게 설명하면 초점이 흐려진다
- **주어를 "당신"으로 시작한다**: Claude Code에게 직접 역할을 부여하는 형식

#### 섹션 2: 핵심 역할

이 에이전트가 수행하는 구체적인 업무를 나열한다.

```markdown
## 핵심 역할
1. **코드 품질 검증**: 가독성, 유지보수성, 성능 관점에서 코드 검토
2. **버그 탐지**: 논리 오류, 엣지 케이스 누락, 잠재적 런타임 에러 식별
3. **개선 제안**: 리팩토링 방향, 디자인 패턴 적용, 테스트 보완 제안
4. **보안 점검**: SQL 인젝션, XSS 등 보안 취약점 검사
```

작성 원칙:

- **번호를 매긴다**: 번호는 우선순위를 암시한다. 1번이 가장 핵심적인 역할이다.
- **3~5개로 제한한다**: 역할이 많아지면 에이전트가 집중하지 못한다. 6개 이상 나열해야 한다면 에이전트를 분리하는 신호다.
- **각 항목을 "명사: 설명" 형태**로 쓴다: 볼드 처리된 명사가 역할의 이름이 되고, 뒤의 설명이 범위를 한정한다.
- **검증 가능한 표현**을 쓴다: "좋은 코드를 만든다"(검증 불가)보다 "가독성, 유지보수성 관점에서 검토"(검증 가능)가 낫다.

#### 섹션 3: 작업 원칙

모든 출력에 적용되는 행동 규칙이다.

```markdown
## 작업 원칙
- Always respond in Korean
- 심각도를 3단계로 분류: CRITICAL / WARNING / SUGGESTION
- 문제 지적 시 반드시 **구체적인 수정 코드**를 함께 제시
- 좋은 부분도 언급하여 균형 잡힌 리뷰 유지
- 추측이나 가정을 하지 않음 — 확인이 필요하면 질문
```

핵심 역할이 "무엇을 하는가"라면, 작업 원칙은 "어떤 기준으로 하는가"다. 작업 원칙은 에이전트의 출력 품질을 결정하는 가장 중요한 섹션이다.

좋은 작업 원칙의 특징:

| 특징 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| 구체적 | "깔끔하게 리뷰" | "심각도를 CRITICAL/WARNING/SUGGESTION 3단계로 분류" |
| 검증 가능 | "친절하게 설명" | "문제 지적 시 수정 코드를 함께 제시" |
| 행동 지향 | "코드를 잘 이해" | "확인이 필요하면 질문" |
| 제한 범위 | "모든 것을 검토" | "가독성, 유지보수성, 성능 관점에서 검토" |

작업 원칙은 5~8개가 적당하다. 너무 적으면 에이전트의 행동이 예측 불가능해지고, 너무 많으면 원칙 간 충돌이 발생한다.

#### 섹션 4: 출력 형식

에이전트의 산출물이 일정한 형태를 가져야 할 때 정의한다.

```markdown
## 출력 형식

```markdown
## 리뷰: {파일명 또는 PR 제목}

### 총평
- 코드 품질: {상/중/하}
- 주요 이슈: {count}개

### 상세 리뷰

#### [CRITICAL] {위치 — 파일:라인}
- 문제: {설명}
- 수정 전: `{기존 코드}`
- 수정 후: `{제안 코드}`

#### [WARNING] {위치}
- 문제: {설명}
- 제안: {개선 방향}

#### [SUGGESTION] {위치}
- 제안: {개선 아이디어}

### 잘된 점
- {긍정적 피드백}
```

출력 형식이 필요한 경우와 필요하지 않은 경우:

- **필요한 경우**: 다른 에이전트가 이 출력을 파싱해서 사용할 때, 사용자가 일관된 형태를 기대할 때
- **필요하지 않은 경우**: 자유 형식의 텍스트를 생성할 때, 출력 형태가 매번 달라질 때

출력 형식을 정의할 때는 **템플릿을 마크다운 코드 블록으로 감싸서** 보여준다. 이렇게 하면 에이전트가 형식을 정확히 따른다.

> **참고: 코드 블록 안에 코드 블록을 넣는 방법.** 마크다운 파일 안에서 코드 블록(` ``` `) 안에 또 다른 코드 블록을 넣어야 할 때가 있다. 이 경우 내부 코드 블록의 ` ``` ` 앞에 zero-width space(U+200B, 폭이 0인 보이지 않는 문자)를 삽입하면, 외부 코드 블록이 닫히지 않고 중첩이 가능하다. 이 책의 예제에서도 이 기법이 사용된다.

#### 섹션 5: 협업

팀의 일원으로 동작할 때, 다른 에이전트와의 관계를 정의한다.

```markdown
## 협업
- `chapter-writer`가 작성한 코드를 입력으로 받음
- 리뷰 결과를 `chapter-writer`에게 반환하여 수정 요청
- 수정본을 재리뷰하여 승인/재수정 판단
```

협업 섹션은 세 가지를 명시한다:

1. **입력**: 누구에게서 무엇을 받는가
2. **출력**: 누구에게 무엇을 전달하는가
3. **상호작용 패턴**: 한 번으로 끝나는가, 루프를 도는가

단독으로 사용되는 에이전트라면 이 섹션을 생략해도 된다. 하지만 하네스의 일부로 동작하는 에이전트라면 반드시 포함해야 한다. 협업 섹션이 없는 에이전트는 팀에서 자기 위치를 모르는 팀원과 같다.

### 실습: "코드 리뷰어" 에이전트를 step-by-step으로 작성

이제 배운 내용을 적용하여 실제 에이전트를 처음부터 작성해보자. 목표는 코드 리뷰를 전문으로 수행하는 에이전트를 만드는 것이다.

#### Step 1: 역할 정의 — 이 에이전트는 누구인가?

에이전트를 작성하기 전에, 먼저 다음 질문에 답한다:

- **무엇을 하는 에이전트인가?** → 코드의 품질을 검증하고 개선안을 제시한다
- **어떤 전문성이 필요한가?** → 가독성, 버그 탐지, 보안, 성능 분석
- **누구와 협업하는가?** → 코드를 작성하는 개발자 에이전트에게서 입력을 받고, 리뷰 결과를 돌려준다
- **어떤 형태로 결과를 내는가?** → 심각도별로 분류된 구조화된 리뷰

이 답변들이 에이전트 파일의 각 섹션에 대응된다.

#### Step 2: frontmatter 작성

```yaml
---
name: code-reviewer
description: "코드 리뷰를 수행하는 에이전트. '코드 리뷰', 'code review', 'PR 리뷰', '코드 검토' 요청 시 트리거."
model: sonnet
---
```

각 필드의 선택 이유:

- `name: code-reviewer` — 역할이 명확하고 kebab-case를 따른다
- `description` — 역할 설명 한 문장 + 트리거 키워드 4개. 한국어("코드 리뷰", "코드 검토")와 영어("code review")를 모두 포함했다
- `model: sonnet` — 코드 리뷰는 체크리스트 기반 분석 작업이다. 복잡한 창작이 아니므로 sonnet이면 충분하다

#### Step 3: 역할 선언

```markdown
# Code Reviewer — 코드 리뷰 전문가

당신은 소프트웨어 프로젝트의 코드 품질을 검증하는 전문 리뷰어입니다.
```

"소프트웨어 프로젝트의 코드 품질을 검증하는"이라는 수식어가 범위를 한정한다. 단순히 "리뷰어"가 아니라 "코드 품질 검증"이라는 구체적 영역을 명시했다.

#### Step 4: 핵심 역할 정의

```markdown
## 핵심 역할
1. **코드 품질 검증**: 가독성, 유지보수성, 성능 관점에서 코드 검토
2. **버그 탐지**: 논리 오류, 엣지 케이스 누락, 잠재적 런타임 에러 식별
3. **개선 제안**: 리팩토링 방향, 디자인 패턴 적용, 테스트 보완 제안
4. **보안 점검**: SQL 인젝션, XSS, 인증/인가 누락 등 보안 취약점 검사
```

4개의 역할이며, 우선순위 순서로 나열했다. 코드 품질 검증이 가장 기본이고, 보안 점검은 해당되는 경우에만 수행하므로 마지막이다.

#### Step 5: 작업 원칙 작성

```markdown
## 작업 원칙
- Always respond in Korean
- 심각도를 3단계로 분류: CRITICAL / WARNING / SUGGESTION

7개의 원칙이다. 각각이 구체적이고 검증 가능하다. "심각도를 3단계로 분류"는 출력 형식과 연동되며, "코딩 스타일보다 로직과 설계에 집중"은 리뷰의 방향을 명확히 한다.

# Code Reviewer — 코드 리뷰 전문가

당신은 소프트웨어 프로젝트의 코드 품질을 검증하는 전문 리뷰어입니다.

## 핵심 역할
1. **코드 품질 검증**: 가독성, 유지보수성, 성능 관점에서 코드 검토
2. **버그 탐지**: 논리 오류, 엣지 케이스 누락, 잠재적 런타임 에러 식별
3. **개선 제안**: 리팩토링 방향, 디자인 패턴 적용, 테스트 보완 제안
4. **보안 점검**: SQL 인젝션, XSS, 인증/인가 누락 등 보안 취약점 검사

## 작업 원칙
- Always respond in Korean
- 심각도를 3단계로 분류: CRITICAL / WARNING / SUGGESTION
- 문제 지적 시 반드시 **구체적인 수정 코드**를 함께 제시
- 좋은 부분도 언급하여 균형 잡힌 리뷰 유지
- 추측이나 가정을 하지 않음 — 확인이 필요하면 질문
- 한 번에 하나의 파일 또는 PR 단위로 리뷰
- 코딩 스타일보다 **로직과 설계**에 집중

## 출력 형식

리뷰 결과는 다음 형식을 따릅니다:

​```markdown
## 리뷰: {파일명 또는 PR 제목}

### 총평
- 코드 품질: {상/중/하}
- 주요 이슈: {count}개

### 상세 리뷰

#### [CRITICAL] {위치 — 파일:라인}
- 문제: {설명}
- 수정 전: `{기존 코드}`
- 수정 후: `{제안 코드}`

#### [WARNING] {위치}
- 문제: {설명}
- 제안: {개선 방향}

#### [SUGGESTION] {위치}
- 제안: {개선 아이디어}

### 잘된 점
- {긍정적 피드백}
​```

## 협업
- 개발자 에이전트가 작성한 코드를 입력으로 받음
- 리뷰 결과를 개발자 에이전트에게 반환하여 수정 요청
- CRITICAL 이슈가 있으면 수정본을 재리뷰
- CRITICAL 이슈가 0개일 때 승인
```

총 40줄 남짓이다. 에이전트 파일은 길 필요가 없다. 핵심만 명확하게 담으면 된다.

### 심화: description에 트리거 키워드를 효과적으로 넣는 법

`description` 필드는 에이전트의 "검색 가능성"을 결정한다. 사용자가 어떤 표현으로 요청하든 이 에이전트가 선택되게 하려면, 키워드 설계에 신경 써야 한다.

#### 원칙 1: 사용자의 언어로 쓴다

개발자가 "코드 리뷰해줘"라고 하지 "소스코드 정적 분석을 수행해줘"라고 하지 않는다. 트리거 키워드는 사용자가 **실제로 입력할 표현**이어야 한다.

```yaml
# 나쁜 예 — 전문 용어 위주
description: "소스코드 정적 분석 및 동적 검증 에이전트. '정적 분석', 'static analysis' 요청 시 트리거."

# 좋은 예 — 사용자 표현 위주
description: "코드 리뷰를 수행하는 에이전트. '코드 리뷰', 'code review', 'PR 리뷰', '코드 검토' 요청 시 트리거."
```

#### 원칙 2: 한국어와 영어를 모두 포함한다

개발자는 한국어와 영어를 섞어 쓴다. 두 언어의 키워드를 모두 포함해야 트리거 누락을 방지한다.

```yaml
description: "... '코드 리뷰', 'code review', 'PR 리뷰' ..."
```

#### 원칙 3: 동의어와 유사 표현을 포함한다

같은 의도를 다른 단어로 표현할 수 있다. 주요 동의어를 커버한다.

```yaml
# "리뷰"와 "검토"는 같은 의도
description: "... '코드 리뷰', '코드 검토', 'code review' ..."
```

다만 키워드를 너무 많이 넣으면 역효과가 난다. 관련 없는 요청에도 매칭될 수 있다. 핵심 키워드 3~5개가 적당하다.

#### 원칙 4: 역할 설명 문장 자체에도 키워드를 녹인다

트리거 키워드 목록뿐 아니라, 앞의 역할 설명 문장에도 핵심 단어를 포함시킨다. Claude Code는 description 전체를 매칭에 사용하기 때문이다.

```yaml
# 역할 설명에 "코드 리뷰"가 이미 포함됨
description: "코드 리뷰를 수행하는 에이전트. '코드 리뷰', 'code review', 'PR 리뷰', '코드 검토' 요청 시 트리거."
```

#### 실전 예시 비교

아래 표는 다양한 에이전트의 description을 보여준다. 이 책의 실제 에이전트들에서 가져온 패턴이다.

| 에이전트 | description |
|---------|-------------|
| `book-planner` | "책의 목차와 구조를 설계하는 에이전트. '목차 설계', '책 구조', 'book plan' 요청 시 트리거." |
| `chapter-writer` | "책의 챕터를 집필하는 에이전트. '챕터 작성', '집필', 'write chapter' 요청 시 트리거." |
| `book-editor` | "원고를 리뷰하고 편집하는 에이전트. '원고 리뷰', '편집', 'review chapter' 요청 시 트리거." |

공통 패턴이 보인다: **"{역할 동사}하는 에이전트. '{한국어 키워드}', '{영어 키워드}' 요청 시 트리거."** 이 패턴을 따르면 일관성 있는 description을 작성할 수 있다.

### 심화: model 선택 가이드 — sonnet vs opus

모델 선택은 비용과 품질 사이의 트레이드오프다. 잘못된 선택은 불필요한 비용 증가 또는 품질 저하로 이어진다.

#### 판단 기준

다음 표는 작업 특성에 따른 모델 선택 기준이다.

| 작업 특성 | sonnet | opus |
|----------|--------|------|
| 구조화된 분석 (체크리스트 기반) | O | |
| 패턴 기반 코드 생성 | O | |
| 데이터 변환/포맷팅 | O | |
| 짧은 텍스트 생성 (요약, 설명) | O | |
| 긴 텍스트 창작 (챕터, 보고서) | | O |
| 복잡한 추론 (다단계 논리) | | O |
| 미묘한 판단 (뉘앙스, 맥락) | | O |
| 여러 제약 조건 동시 만족 | | O |

이 기준을 한 문장으로 요약하면: **"정해진 틀에 따라 처리하는 작업은 sonnet, 틀 자체를 만들어야 하는 작업은 opus."**

#### 실전 적용: book-writer 하네스의 모델 배정

이 책의 에이전트 팀에서 왜 `chapter-writer`만 opus이고 나머지는 sonnet인지 살펴보자.

| 에이전트 | 모델 | 이유 |
|---------|------|------|
| `book-planner` | sonnet | 목차 구조화는 분석적 작업. 주어진 주제를 분류하고 순서를 정하면 된다 |
| `chapter-writer` | opus | 12~18페이지의 본문을 일관된 문체로 작성해야 한다. 개념 설명, 비유, 예제를 유기적으로 엮는 창작 작업이다 |
| `book-editor` | sonnet | 체크리스트 기반 검증이다. "정확한가? 완성도는? 일관성은?"을 항목별로 확인한다 |
| `infographic-creator` | sonnet | SVG 코드 생성은 패턴 기반이다. 박스, 화살표, 텍스트의 조합이다 |
| `book-publisher` | sonnet | PDF/웹 변환은 절차적 작업이다. 정해진 규칙에 따라 처리한다 |

#### 비용 최적화 전략

모델 선택에 확신이 없다면, 다음 전략을 추천한다:

1. **모든 에이전트를 sonnet으로 시작한다**
2. **결과 품질이 부족한 에이전트만 opus로 올린다**
3. **opus로 올린 후에도 개선이 미미하면 sonnet으로 돌린다** — 이 경우 문제는 모델이 아니라 에이전트 정의 자체에 있다

세 번째 포인트가 중요하다. 품질 문제의 원인이 항상 모델 성능인 것은 아니다. 핵심 역할이 모호하거나, 작업 원칙이 부족하거나, 출력 형식이 불명확한 경우 opus로 올려도 개선되지 않는다. **모델을 바꾸기 전에 에이전트 정의를 먼저 점검하라.**

같은 역할의 에이전트를 두 가지 방식으로 작성해보겠다. 차이를 비교하면서 좋은 정의의 조건을 정리하자.

#### 나쁜 에이전트 정의

name: code-reviewer

#### 차이점 분석

| 항목 | 나쁜 정의 | 좋은 정의 | 왜 중요한가 |
|------|----------|----------|------------|
| name | `reviewer` | `code-reviewer` | 범위가 모호하면 잘못된 상황에서 호출된다 |
| description | 트리거 키워드 없음 | 한/영 키워드 4개 | 키워드 없으면 사용자 요청에 매칭되지 않는다 |
| model | `opus` | `sonnet` | 분석 작업에 opus는 비용 낭비다 |
| 역할 선언 | "리뷰를 잘 해주세요" | "코드 품질을 검증하는 전문 리뷰어" | 페르소나가 모호하면 관점이 흔들린다 |

| 출력 형식 | 없음 | 구조화된 템플릿 | 형식 없으면 매번 다른 형태가 나온다 |
| 협업 | 없음 | 입력/출력/종료 조건 명시 | 팀에서 제 역할을 못 한다 |

나쁜 정의의 핵심 문제는 **추상성**이다. "잘 해주세요", "꼼꼼하게", "친절하게"는 사람이 들어도 어떻게 행동해야 할지 모호하다. AI도 마찬가지다. 구체적이고 검증 가능한 지시가 일관된 품질을 만든다.

좋은 에이전트 정의의 조건을 정리하면 다음과 같다:

1. **name이 역할을 드러낸다** — 파일명만 봐도 무슨 에이전트인지 안다
2. **description에 트리거 키워드가 있다** — 사용자의 자연스러운 표현과 매칭된다
3. **model이 작업 특성에 맞다** — 비용과 품질의 균형
4. **역할 선언이 구체적이다** — "당신은 ~하는 전문가입니다"로 범위를 한정한다
5. **핵심 역할이 3~5개로 제한된다** — 집중할 수 있는 범위
6. **작업 원칙이 검증 가능하다** — "잘 하세요"가 아니라 "3단계로 분류"
7. **출력 형식이 정의되어 있다** — 일관된 산출물 보장
8. **협업 관계가 명시되어 있다** — 팀 내 위치와 종료 조건

---

## 정리

- 에이전트 파일은 **frontmatter**(식별/라우팅)와 **본문**(행동 지침) 두 영역으로 구성된다.
- frontmatter의 세 필드: `name`(고유 ID), `description`(역할 + 트리거 키워드), `model`(AI 모델).
- 본문의 핵심 섹션: 역할 선언, 핵심 역할(3~5개), 작업 원칙(5~8개), 출력 형식, 협업.
- 트리거 키워드는 **사용자의 언어**로, **한국어/영어 모두** 포함하되 **3~5개**로 제한한다.
- 모델 선택: 정해진 틀에 따라 처리하는 작업은 **sonnet**, 틀 자체를 만드는 작업은 **opus**. 확신이 없으면 sonnet으로 시작하여 점진적으로 올린다.
- 좋은 에이전트 정의의 핵심은 **구체성과 검증 가능성**이다. "잘 해주세요"가 아니라 "심각도를 3단계로 분류"처럼 쓴다.
