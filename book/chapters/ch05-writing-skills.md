# Chapter 5: 스킬 정의 작성하기

## 이 챕터에서 배울 것

- 스킬 파일(`.claude/skills/*/SKILL.md`)의 전체 구조와 각 섹션의 역할
- frontmatter, 워크플로우, 도구 사용법, 출력 규칙을 올바르게 작성하는 방법
- 에이전트 1개 : 스킬 N개의 관계를 설계하는 원칙
- 여러 에이전트가 하나의 스킬을 공유하는 구조 설계
- `references/` 디렉토리로 큰 참고자료를 분리하는 기법
- 워크플로우의 Phase별 검증 기준이 왜 중요한지, 어떻게 명시하는지
- Chapter 4에서 만든 "코드 리뷰어" 에이전트와 연결되는 "코드 리뷰" 스킬을 직접 작성

## 본문

### 스킬 파일이란 무엇인가

Chapter 1에서 스킬은 "어떻게 하는가"를 정의하는 파일이라고 배웠다. 에이전트가 "누구인가"를 정의한다면, 스킬은 그 에이전트가 **따라야 할 구체적인 작업 절차**를 담는다.

에이전트 파일이 사람의 이력서라면, 스킬 파일은 **업무 매뉴얼**이다. 이력서에는 "나는 편집자입니다"라고 쓰지만, 업무 매뉴얼에는 "원고를 받으면 1단계로 구조를 검토하고, 2단계로 문법을 확인하고..."라는 절차가 적혀 있다.

스킬 파일은 `.claude/skills/{스킬명}/SKILL.md` 경로에 위치한다. 에이전트와 달리 **디렉토리 단위**로 관리된다는 점이 중요하다.

```
.claude/skills/
└── code-review/           # 스킬 디렉토리
    ├── SKILL.md           # 스킬 정의 본체
    └── references/        # 참고자료 (선택)
        ├── checklist.md
        └── severity-guide.md
```

왜 디렉토리일까? 스킬에는 워크플로우 외에도 체크리스트, 템플릿, 예시 파일 같은 참고자료가 필요할 수 있기 때문이다. 이들을 하나의 디렉토리에 묶으면 스킬 단위로 이동하거나 재사용하기 쉽다.

---

### 스킬 파일의 전체 구조

스킬 파일은 크게 4개 섹션으로 구성된다.

```markdown
---
name: code-review
description: "코드 리뷰 스킬. PR의 코드를 분석하고 리뷰 코멘트를 생성합니다.
              '코드 리뷰', 'code review', 'PR 리뷰' 요청 시 트리거."
---

# 1. 개요 (Overview)
이 스킬이 무엇을 하는지 한 문단 요약.

# 2. 워크플로우 (Workflow)
Phase별 작업 절차. 입력 → 처리 → 출력 → 검증.

# 3. 도구 사용법 (Tool Usage)
각 Phase에서 사용하는 Claude Code 도구와 사용 규칙.

# 4. 출력 규칙 (Output Rules)
산출물의 형식, 저장 위치, 네이밍 규칙.
```

각 섹션을 하나씩 살펴보자.

#### 섹션 1: frontmatter

스킬 파일의 frontmatter는 에이전트와 동일한 형식이지만, **`model` 필드가 없다**.

```yaml
---
name: code-review
description: "코드 리뷰 스킬. PR의 코드를 분석하고 리뷰 코멘트를 생성합니다.
              '코드 리뷰', 'code review', 'PR 리뷰' 요청 시 트리거."
---
```

- **`name`**: 스킬의 고유 식별자. 에이전트 파일이나 오케스트레이터에서 이 이름으로 참조한다.
- **`description`**: 스킬의 역할 설명 + 트리거 키워드. 사용자가 "코드 리뷰해줘"라고 말하면, Claude Code가 이 description을 보고 해당 스킬을 활성화한다.

왜 `model`이 없는가? 스킬은 절차를 정의할 뿐, 어떤 모델로 실행할지는 스킬을 호출하는 에이전트가 결정하기 때문이다. 같은 "코드 리뷰" 스킬을 `opus` 에이전트가 실행할 수도 있고 `sonnet` 에이전트가 실행할 수도 있다. 절차는 같지만 실행 모델은 다를 수 있다.

#### 섹션 2: 워크플로우

워크플로우는 스킬의 핵심이다. "무엇을 어떤 순서로 하는가"를 Phase 단위로 정의한다.

```markdown
## 워크플로우

### Phase 1: 변경 파일 수집
- **입력**: PR 번호 또는 브랜치명
- **처리**: `gh pr diff` 또는 `git diff`로 변경된 파일 목록과 diff 수집
- **출력**: 변경 파일 목록 + diff 내용
- **검증**: diff가 비어있지 않은지 확인

### Phase 2: 코드 분석
- **입력**: Phase 1의 diff 내용
- **처리**: 파일별로 변경 사항 분석, 이슈 식별
- **출력**: 파일별 이슈 목록 (심각도, 위치, 설명)
- **검증**: 모든 변경 파일이 분석되었는지 확인
```

각 Phase에는 반드시 4가지를 명시한다:

| 항목 | 역할 | 예시 |
|------|------|------|
| **입력** | 이 Phase가 시작되려면 무엇이 필요한가 | PR 번호, diff 내용 |
| **처리** | 입력을 어떻게 가공하는가 | diff를 파싱하여 이슈 식별 |
| **출력** | 이 Phase가 끝나면 무엇이 생기는가 | 이슈 목록, 리뷰 코멘트 |
| **검증** | 출력이 올바른지 어떻게 확인하는가 | 모든 파일이 분석되었는지 |

이 4가지 중 하나라도 빠지면 스킬은 불완전하다. 특히 **검증**이 빠지는 경우가 많은데, 이것이 왜 문제인지는 뒤의 심화 섹션에서 다룬다.

#### 섹션 3: 도구 사용법

Claude Code는 파일 읽기, 검색, 편집, 터미널 명령 실행 등 여러 도구를 가지고 있다. 스킬 파일에서는 각 Phase에서 **어떤 도구를 어떻게 사용해야 하는지** 명시한다.

```markdown
## 도구 사용법

### 파일 변경 확인
- `git diff main...HEAD` 로 변경된 파일의 diff를 확인한다
- `gh pr view {번호} --json files` 로 PR의 변경 파일 목록을 가져온다

### 코드 읽기
- Read 도구로 변경된 파일의 전체 내용을 읽는다
- 변경된 줄 주변의 맥락을 이해하기 위해 앞뒤 50줄을 함께 읽는다

### 리뷰 코멘트 작성
- `gh pr review {번호} --comment --body "..."` 로 리뷰 코멘트를 남긴다
- 인라인 코멘트가 필요하면 `gh api` 를 사용한다
```

왜 이것을 명시해야 할까? Claude Code는 같은 작업을 여러 방법으로 수행할 수 있다. 파일을 읽을 때 `Read` 도구를 쓸 수도 있고, `Bash`에서 `cat`을 실행할 수도 있다. 스킬 파일에서 도구를 지정하면 **일관된 방식**으로 작업이 수행되고, 결과의 예측 가능성이 높아진다.

#### 섹션 4: 출력 규칙

스킬의 최종 산출물이 어떤 형식이어야 하고, 어디에 저장되는지를 정의한다.

```markdown
## 출력 규칙

### 리뷰 결과 형식
- 이슈별로 심각도(CRITICAL, WARNING, SUGGESTION) 태그를 붙인다
- 각 이슈에는 파일 경로, 라인 번호, 설명, 개선 제안을 포함한다
- 리뷰 요약은 마크다운 테이블로 작성한다

### 산출물 저장
- 리뷰 결과는 `reviews/review-{PR번호}.md` 에 저장한다
- 파일명은 소문자, 하이픈 구분자를 사용한다
```

출력 규칙이 없으면 같은 스킬을 실행할 때마다 형식이 달라진다. 한 번은 테이블로, 한 번은 리스트로, 한 번은 자유 서술로 결과가 나올 수 있다. 출력 규칙은 이 **비결정성을 줄여준다**.

### 에이전트와 스킬의 관계 설계

#### 1 에이전트 : N 스킬

하나의 에이전트가 여러 스킬을 참조할 수 있다. 이것이 가장 흔한 구조다.

```
.claude/
├── agents/
│   └── code-reviewer.md       # 코드 리뷰어 에이전트
└── skills/
    ├── code-review/           # 코드 리뷰 스킬
    │   └── SKILL.md
    ├── security-audit/        # 보안 감사 스킬
    │   └── SKILL.md
    └── performance-review/    # 성능 리뷰 스킬
        └── SKILL.md
```

`code-reviewer` 에이전트는 상황에 따라 세 가지 스킬 중 하나를 선택하여 실행한다. 일반 코드 리뷰를 요청하면 `code-review` 스킬을, 보안 취약점 검토를 요청하면 `security-audit` 스킬을 사용한다.

왜 이렇게 설계할까? 에이전트의 **전문성은 동일**(코드를 분석하는 능력)하지만, **절차가 다르기** 때문이다. 보안 감사는 일반 코드 리뷰와 체크리스트가 다르고, 검증 기준도 다르다. 에이전트를 3개로 나누는 것보다 스킬을 3개로 나누는 것이 합리적이다.

이 관계는 에이전트 파일에서 다음과 같이 표현할 수 있다:

```markdown
## 사용 가능한 스킬
- `code-review`: 일반 코드 리뷰 (기본)
- `security-audit`: 보안 취약점 중심 리뷰
- `performance-review`: 성능 병목 중심 리뷰

사용자 요청에 따라 적절한 스킬을 선택하여 실행합니다.
```

#### N 에이전트 : 1 스킬 (공유 스킬)

반대로, 여러 에이전트가 **하나의 스킬을 공유**할 수도 있다.

```
.claude/
├── agents/
│   ├── backend-dev.md         # 백엔드 개발자
│   ├── frontend-dev.md        # 프론트엔드 개발자
│   └── devops-engineer.md     # DevOps 엔지니어
└── skills/
    └── code-review/           # 공유되는 코드 리뷰 스킬
        └── SKILL.md
```

세 에이전트 모두 PR을 리뷰할 수 있고, 리뷰 절차는 동일하다. 차이는 관점이다. 백엔드 개발자는 API 설계를 중점적으로 보고, 프론트엔드 개발자는 UI 로직을 중점적으로 본다. 이 관점 차이는 에이전트 파일에 정의되어 있으므로, 절차(스킬)는 공유해도 된다.

공유 스킬을 설계할 때의 핵심 원칙은 **절차는 일반적으로, 관점은 에이전트에게 위임**하는 것이다. 스킬에 "API 설계를 중점적으로 본다"라고 쓰면 공유가 불가능해진다. 대신 "변경된 파일을 에이전트의 전문 분야 관점에서 분석한다"처럼 일반적으로 작성한다.

#### 설계 판단 기준

에이전트와 스킬의 관계를 결정할 때 다음 질문을 던져보자:

| 질문 | 답이 "예"이면 |
|------|-------------|
| 같은 에이전트가 상황에 따라 다른 절차를 따르는가? | 1 에이전트 : N 스킬 |
| 다른 에이전트가 동일한 절차를 따르는가? | N 에이전트 : 1 스킬 |
| 에이전트마다 절차도 다르고 관점도 다른가? | 1 에이전트 : 1 스킬 |

### references/ 디렉토리 활용

스킬의 워크플로우를 작성하다 보면 참고자료가 길어지는 경우가 있다. 코드 리뷰 체크리스트가 100줄이 넘거나, 보안 취약점 목록이 별도 문서로 존재하거나, 외부 가이드라인을 포함해야 하는 경우다.

이런 참고자료를 SKILL.md 안에 모두 넣으면 두 가지 문제가 생긴다:

1. **가독성 저하**: 워크플로우의 흐름 사이에 100줄짜리 체크리스트가 끼어 있으면 전체 구조를 파악하기 어렵다.
2. **컨텍스트 낭비**: 특정 Phase에서만 필요한 참고자료가 전체 실행 내내 컨텍스트를 차지한다.

해결책은 `references/` 디렉토리로 분리하고, SKILL.md에서 필요할 때만 참조하는 것이다.

#### 분리 전: 모든 것이 SKILL.md에

```markdown
## 워크플로우

### Phase 2: 코드 분석
- 다음 체크리스트에 따라 코드를 분석한다:

#### 체크리스트
1. 함수가 단일 책임 원칙을 따르는가
2. 변수명이 의미를 명확히 전달하는가
3. 에러 처리가 적절한가
4. 중복 코드가 없는가
5. 테스트가 포함되어 있는가
... (50줄 이상 계속)

#### 심각도 기준
- CRITICAL: 프로덕션 장애 가능성
- WARNING: 유지보수 비용 증가
- SUGGESTION: 개선 제안
... (30줄 이상 계속)

### Phase 3: 리뷰 코멘트 작성
```

워크플로우의 흐름이 체크리스트에 의해 끊긴다.

#### 분리 후: references/로 추출

**SKILL.md:**

```markdown
## 워크플로우

### Phase 2: 코드 분석
- `references/checklist.md` 의 체크리스트에 따라 코드를 분석한다
- 심각도는 `references/severity-guide.md` 의 기준을 따른다

### Phase 3: 리뷰 코멘트 작성
```

**references/checklist.md:**

```markdown
# 코드 리뷰 체크리스트

## 구조
1. 함수가 단일 책임 원칙을 따르는가
2. 클래스/모듈의 응집도가 높은가
3. 의존성 방향이 올바른가

## 가독성
4. 변수명이 의미를 명확히 전달하는가
5. 함수명이 동작을 정확히 설명하는가
6. 복잡한 로직에 주석이 있는가

## 안정성
7. 에러 처리가 적절한가
8. 엣지 케이스가 고려되었는가
9. null/undefined 처리가 되어 있는가

## 테스트
10. 변경에 대한 테스트가 포함되어 있는가
11. 테스트가 경계 조건을 다루는가
12. 테스트 이름이 의도를 설명하는가
```

**references/severity-guide.md:**

```markdown
# 심각도 분류 기준

## CRITICAL
프로덕션 환경에서 장애를 일으킬 수 있는 이슈. 머지 전 반드시 수정해야 한다.
- 보안 취약점 (SQL 인젝션, XSS 등)
- 데이터 손실 가능성
- 무한 루프 또는 데드락
- 인증/인가 우회

## WARNING
즉시 장애는 아니지만 유지보수 비용을 높이는 이슈. 수정을 권장한다.
- 중복 코드
- 과도한 복잡도 (순환 복잡도 10 초과)
- 누락된 에러 처리
- 하드코딩된 설정값

## SUGGESTION
코드 품질 개선을 위한 제안. 수정 여부는 작성자 판단에 맡긴다.
- 네이밍 개선 제안
- 더 나은 API 활용법
- 주석 추가 권장
- 리팩토링 기회
```

이 구조의 장점은 세 가지다:

1. **SKILL.md의 가독성 유지**: 워크플로우의 흐름이 깔끔하게 읽힌다.
2. **참고자료의 독립적 관리**: 체크리스트만 업데이트하고 싶을 때 SKILL.md를 건드리지 않아도 된다.
3. **선택적 로딩**: 에이전트가 Phase 2를 실행할 때만 `references/checklist.md`를 읽으면 된다. 컨텍스트를 절약할 수 있다.

#### references/ 분리 판단 기준

모든 참고자료를 분리할 필요는 없다. 다음 기준으로 판단한다:

| 기준 | 분리 | SKILL.md에 유지 |
|------|------|----------------|
| 분량 | 20줄 이상 | 20줄 미만 |
| 변경 빈도 | 자주 업데이트됨 | 거의 바뀌지 않음 |
| 사용 범위 | 특정 Phase에서만 사용 | 전체 워크플로우에 걸쳐 사용 |
| 공유 가능성 | 다른 스킬에서도 재사용 | 이 스킬에서만 사용 |

이 책의 프로젝트에서도 `references/` 디렉토리를 활용하고 있다. Karpathy의 LLM 코딩 가이드라인은 `references/karpathy-guidelines/`에, Kent Beck의 TDD 원칙은 `references/kentbeck-plan-driven-tdd/`에 분리되어 있다. 이들은 프로젝트 전체의 코딩 원칙으로 참조되지만, CLAUDE.md에 전문을 넣으면 컨텍스트를 과도하게 차지하므로 별도 디렉토리로 분리한 것이다.

```markdown
# Code Reviewer — 코드 리뷰 전문가

당신은 소프트웨어 프로젝트의 코드 품질을 검증하는 전문 리뷰어입니다.

## 핵심 역할
1. **코드 분석**: 변경된 코드의 구조, 가독성, 안정성을 평가
2. **이슈 식별**: 버그, 보안 취약점, 성능 문제를 발견
3. **개선 제안**: 구체적이고 실행 가능한 개선 방안을 제시
4. **교육적 피드백**: 왜 그렇게 바꿔야 하는지 이유를 설명

## 작업 원칙
- Always respond in Korean
- 비판이 아닌 건설적 피드백
- 코드 작성자의 의도를 먼저 이해한 뒤 리뷰
- 사소한 스타일 이슈보다 구조적 문제에 집중

## 협업
- 오케스트레이터 또는 사용자로부터 PR 번호를 입력받음
- `code-review` 스킬의 워크플로우를 따라 리뷰 수행
- 리뷰 결과를 사용자 또는 다음 에이전트에게 전달
```

이제 이 에이전트가 따를 스킬 파일을 만든다.

#### Step 1: 디렉토리 생성

```bash
mkdir -p .claude/skills/code-review/references
```

스킬 디렉토리와 `references/` 를 함께 만든다. 참고자료가 필요하지 않더라도 디렉토리를 미리 만들어 두면, 나중에 체크리스트나 가이드를 추가할 때 편하다.

#### Step 2: frontmatter 작성

`.claude/skills/code-review/SKILL.md`를 만들고, frontmatter부터 작성한다:

포인트:

```markdown
# Code Review — 코드 리뷰 스킬

PR(Pull Request)의 변경 사항을 분석하고, 코드 품질 이슈를 식별하여,
구조화된 리뷰 코멘트를 생성하는 스킬.
```

개요는 1~2문장으로 충분하다. 이 스킬이 **무엇을 하는지**만 전달하면 된다.

#### Step 4: 워크플로우 작성

여기가 스킬의 핵심이다. 코드 리뷰의 절차를 Phase로 나눈다.

```markdown
## 워크플로우

### Phase 1: 변경 파일 수집
- **입력**: PR 번호 또는 브랜치명
- **처리**:
  1. `gh pr view {번호} --json files,title,body` 로 PR 정보 수집
  2. `gh pr diff {번호}` 로 전체 diff 수집
  3. 변경된 파일을 유형별로 분류 (소스코드, 테스트, 설정 등)
- **출력**: PR 메타 정보 + 파일별 diff
- **검증**: diff가 비어있지 않은지, 파일 목록이 PR 정보와 일치하는지 확인

### Phase 2: 코드 분석
- **입력**: Phase 1의 diff + 원본 파일
- **처리**:
  1. 변경된 파일별로 Read 도구로 전체 내용을 읽는다
  2. diff의 맥락을 이해하기 위해 변경 지점 전후 코드를 확인한다
  3. `references/checklist.md` 의 항목에 따라 이슈를 식별한다
  4. 각 이슈에 `references/severity-guide.md` 기준으로 심각도를 부여한다
- **출력**: 파일별 이슈 목록 (심각도, 위치, 설명, 개선 제안)
- **검증**: 모든 변경 파일이 분석되었는지, 각 이슈에 심각도가 부여되었는지 확인

### Phase 3: 리뷰 코멘트 생성
- **입력**: Phase 2의 이슈 목록
- **처리**:
  1. 이슈를 심각도순으로 정렬 (CRITICAL → WARNING → SUGGESTION)
  2. 각 이슈에 대해 리뷰 코멘트 작성 (파일, 라인, 설명, 제안)
  3. 전체 리뷰 요약 작성
- **출력**: 구조화된 리뷰 결과 (마크다운)
- **검증**: CRITICAL 이슈가 있으면 요약에 명시되었는지, 모든 이슈에 개선 제안이 포함되었는지 확인

### Phase 4: 리뷰 제출 (선택)
- **입력**: Phase 3의 리뷰 코멘트
- **처리**:
  1. 사용자에게 리뷰 내용을 보여주고 제출 여부를 확인한다
  2. 승인 시 `gh pr review {번호}` 로 리뷰를 제출한다
- **출력**: GitHub PR에 리뷰 코멘트 게시
- **검증**: 리뷰가 정상적으로 제출되었는지 확인
```

각 Phase를 설계할 때 고려한 점:

- **Phase 1과 2를 분리한 이유**: 파일 수집은 도구 호출(git, gh)이고, 코드 분석은 AI의 추론 작업이다. 성격이 다른 작업을 분리하면 오류가 발생했을 때 어느 Phase에서 문제가 생겼는지 특정하기 쉽다.
- **Phase 4가 선택인 이유**: 리뷰 결과를 확인만 하고 싶을 수도 있고, 바로 GitHub에 올리고 싶을 수도 있다. 선택적 Phase로 두면 유연성이 높아진다.
- **모든 Phase에 검증이 있는 이유**: 다음 섹션에서 상세히 다룬다.

#### Step 5: 도구 사용법 작성

```markdown
## 도구 사용법

### GitHub CLI (gh)
- PR 정보 조회: `gh pr view {번호} --json files,title,body,author`
- Diff 수집: `gh pr diff {번호}`
- 리뷰 제출: `gh pr review {번호} --comment --body "{리뷰 내용}"`
- 인라인 코멘트: `gh api repos/{owner}/{repo}/pulls/{번호}/comments`

### 파일 읽기
- Read 도구로 변경된 파일의 전체 내용을 읽는다
- 변경 지점의 맥락 이해를 위해 해당 파일 전체를 읽는다
- 관련 파일(같은 모듈의 다른 파일, 테스트 파일)도 필요하면 함께 읽는다

### 검색
- Grep 도구로 변경된 함수/클래스가 다른 곳에서 사용되는지 확인한다
- Glob 도구로 관련 테스트 파일을 찾는다
```

#### Step 6: 출력 규칙 작성

```markdown

리뷰 결과는 다음 형식의 마크다운으로 작성한다:
```

### 심화: 검증 기준이 없으면 생기는 일

워크플로우의 4가지 요소(입력, 처리, 출력, 검증) 중 가장 자주 빠지는 것이 **검증**이다. "어차피 AI가 알아서 잘 하겠지"라고 생각하기 쉽지만, 검증이 없는 스킬은 세 가지 문제를 일으킨다.

#### 문제 1: 무음 실패

Phase 1에서 `gh pr diff`가 빈 결과를 반환할 수 있다. PR 번호가 잘못되었거나, 아직 커밋이 없는 PR일 수 있다. 검증이 없으면 빈 diff를 가지고 Phase 2가 실행되고, "이슈가 없습니다"라는 거짓 양성 결과가 나온다.

검증이 있으면: "diff가 비어있습니다. PR 번호를 확인해주세요."라고 즉시 알려준다.

#### 문제 2: 오류 누적

Phase 2에서 파일 3개 중 2개만 분석되었다고 하자. Read 도구가 파일 하나를 읽지 못했을 수 있다(경로 변경, 권한 문제 등). 검증이 없으면 Phase 3은 불완전한 이슈 목록으로 리뷰를 작성한다. 결과적으로 리뷰를 받은 개발자는 "나머지 파일은 괜찮은 건가?"라고 혼란에 빠진다.

검증이 있으면: "3개 파일 중 2개만 분석되었습니다. `src/auth/token.py`를 읽지 못했습니다."라고 보고한다.

#### 문제 3: 디버깅 불가

검증이 없는 스킬에서 최종 결과가 이상할 때, 어느 Phase에서 문제가 발생했는지 알 수 없다. 입력이 잘못된 건지, 처리 과정의 버그인지, 출력 형식이 꼬인 건지 추적이 안 된다. Phase마다 검증이 있으면 **오류 지점을 즉시 특정**할 수 있다.

검증 기준을 작성할 때의 원칙:

1. **기계적으로 확인 가능한 조건**을 쓴다. "코드 분석이 충분한지"(주관적)가 아니라 "모든 변경 파일이 분석되었는지"(객관적).
2. **실패 시 행동**을 함께 명시한다. "비어있으면 중단하고 사용자에게 알린다", "누락된 파일이 있으면 목록을 출력하고 계속 진행할지 확인한다" 등.
3. **다음 Phase의 전제 조건**으로 쓴다. Phase 2의 검증은 곧 Phase 3의 입력 조건이다.

### 심화: 스킬 파일 작성의 흔한 실수

스킬 파일을 처음 작성할 때 자주 발생하는 실수와 개선 방법을 정리한다.

#### 실수 1: 워크플로우가 너무 추상적

나쁜 예:

```markdown
### Phase 2: 코드 분석
- 변경된 코드를 분석한다
- 이슈를 찾는다
```

AI에게 "분석하라"고만 하면 매번 다른 방식으로 분석한다. 한 번은 구조를 보고, 한 번은 문법만 보고, 한 번은 성능만 본다.

좋은 예:

```markdown
### Phase 2: 코드 분석
- `references/checklist.md` 의 12개 항목에 대해 순서대로 검토한다
- 각 항목에 대해 해당/비해당/불명확으로 판정한다
- 해당으로 판정된 항목에 대해 이슈를 작성한다
```

**구체적인 기준과 절차**를 명시하면 결과의 일관성이 높아진다.

#### 실수 2: 도구 사용법이 없음

나쁜 예:

```markdown
### Phase 1: 변경 파일 수집
- PR의 변경 사항을 수집한다
```

어떤 도구로 수집하는지 모호하다. `gh pr diff`를 쓸 수도, `git diff`를 쓸 수도, `git log`를 쓸 수도 있다.

좋은 예:

```markdown
### Phase 1: 변경 파일 수집
- `gh pr diff {번호}` 로 diff를 수집한다
- `gh pr view {번호} --json files` 로 변경 파일 목록을 가져온다
```

**정확한 명령어**를 지정하면 실행 결과의 예측 가능성이 높아진다.

#### 실수 3: 출력 형식이 없음

나쁜 예:

```markdown
## 출력 규칙
- 리뷰 결과를 작성한다
```

형식이 없으면 매번 다른 구조로 결과가 나온다. 테이블일 수도, 리스트일 수도, 자유 서술일 수도 있다.

좋은 예:

```markdown
## 출력 규칙
- 요약 섹션: 변경 파일 수, 이슈 수(심각도별), 총평
- 이슈 섹션: 심각도순 정렬, 각 이슈에 파일경로:라인번호, 설명, 제안 포함
- 형식: 마크다운, 이슈 번호는 [C1], [W1], [I1] 등으로 태깅
```

**구체적인 형식 스펙**을 주면 산출물의 일관성이 보장된다.

### 심화: 프로젝트 수준 references와 스킬 수준 references

지금까지 스킬 디렉토리 안의 `references/`를 다뤘다. 하지만 참고자료가 위치할 수 있는 곳은 두 군데다:

```
프로젝트/
├── references/                          # 프로젝트 수준 (전역)
```

**프로젝트 수준 references** (`references/`)는 프로젝트 전체에 적용되는 가이드라인이다. Karpathy의 코딩 원칙, Kent Beck의 TDD 가이드처럼 모든 에이전트와 스킬이 참조할 수 있는 자료다. CLAUDE.md에서 이 파일들을 참조한다.

**스킬 수준 references** (`.claude/skills/{스킬명}/references/`)는 해당 스킬에서만 사용하는 참고자료다. 코드 리뷰 체크리스트는 코드 리뷰 스킬에서만 필요하므로 스킬 디렉토리 안에 둔다.

구분 기준은 단순하다:

| 기준 | 프로젝트 수준 | 스킬 수준 |
|------|-------------|---------|
| 참조 범위 | 여러 스킬/에이전트가 참조 | 해당 스킬만 참조 |
| 예시 | 코딩 원칙, 문체 가이드 | 리뷰 체크리스트, 심각도 기준 |
| 위치 | `references/` | `.claude/skills/{스킬명}/references/` |

## 정리

- 스킬 파일은 4개 섹션으로 구성된다: **frontmatter, 워크플로우, 도구 사용법, 출력 규칙**.
- frontmatter에는 `model` 필드가 없다. 실행 모델은 에이전트가 결정한다.
- 워크플로우의 각 Phase에는 **입력, 처리, 출력, 검증** 4가지를 반드시 명시한다.
- **검증 기준이 없으면** 무음 실패, 오류 누적, 디버깅 불가 문제가 발생한다.
- 에이전트와 스킬의 관계는 **1:N, N:1, 1:1** 모두 가능하다. 절차가 다르면 스킬을 나누고, 관점이 다르면 에이전트를 나눈다.
- 긴 참고자료는 `references/` 디렉토리로 분리한다. SKILL.md의 가독성을 유지하고, 컨텍스트를 절약할 수 있다.
- 참고자료는 프로젝트 수준(`references/`)과 스킬 수준(`.claude/skills/{스킬명}/references/`)으로 나뉜다. 참조 범위에 따라 적절한 위치에 둔다.
