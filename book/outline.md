# 하네스 엔지니어링 실전 가이드

> Claude Code의 Agent + Skill 구조를 활용한 자동화 체계 구축법

## 개요

- 대상 독자: Claude Code 사용자 중 하네스를 처음 구축하려는 개발자. CLI 기본 사용법은 알지만 에이전트 팀을 설계해본 적 없는 수준.
- 목표: 독자가 자신의 도메인에 맞는 하네스(에이전트 팀 + 오케스트레이터 스킬)를 독립적으로 설계하고 구축할 수 있게 한다.
- 총 챕터 수: 12챕터 (4파트)
- 예상 분량: A4 약 150~180페이지
- 산출물: 마크다운 챕터들 + 인포그래픽(SVG) + PDF + 웹 뷰어

---

## 목차

### Chapter 0: 하네스 한눈에 보기
- 목표: 하네스의 핵심 개념을 3분 안에 전달. Agent/Skill을 아는 독자는 이 챕터만 읽고 Part 2로 건너뛸 수 있다
- 핵심 내용:
  - 하네스 = 에이전트 팀 + 오케스트레이터 스킬
  - 실제 동작 예시: book-writer 하네스로 이 책이 만들어진 과정
  - 핵심 아이디어 3가지: 팀 분리, 오케스트레이터 지휘, 마크다운 기반
  - 독자 수준별 추천 읽기 경로
- 분량: 3~4페이지

### Part 1: 기초 — 하네스란 무엇인가

#### Chapter 1: Claude Code의 Agent와 Skill 이해하기
- 목표: Agent와 Skill의 차이를 명확히 구분하고, 각각의 역할과 파일 위치를 설명할 수 있다
- 핵심 내용:
  - Claude Code의 확장 구조 개관 (`.claude/` 디렉토리)
  - Agent: "누가 하는가" — 전문가 페르소나 + 행동 원칙
  - Skill: "어떻게 하는가" — 절차적 가이드 + 도구 번들
  - 파일 위치: `.claude/agents/*.md` vs `.claude/skills/*/SKILL.md`
  - frontmatter 구조 (`name`, `description`, `model`)
  - 트리거 키워드의 역할
- 예제: 간단한 Agent 파일과 Skill 파일을 하나씩 읽어보며 구조 파악
- 인포그래픽: Agent vs Skill 비교 다이어그램
- 분량: 10~12페이지

#### Chapter 2: 하네스의 개념과 구성 요소
- 목표: 하네스가 무엇이며, 왜 필요한지 설명할 수 있다. 하네스의 3요소(에이전트 팀, 오케스트레이터 스킬, 디렉토리 구조)를 이해한다
- 핵심 내용:
  - 하네스 = 에이전트 팀 + 오케스트레이터 스킬을 자동 생성하는 메타 스킬
  - 단일 에이전트의 한계와 팀 구성의 필요성
  - 하네스의 3요소: 에이전트 정의, 스킬 정의, 통합 오케스트레이터
  - 에이전트 분리 기준 4가지: 전문성, 병렬성, 컨텍스트 부담, 재사용성
  - 실제 사례 소개: 이 책 자체가 하네스로 집필됨 (book-writer 하네스)
- 예제: book-writer 하네스의 구조를 도식화하여 살펴보기
- 인포그래픽: 하네스 구성 요소 전체 맵
- 분량: 12~14페이지

#### Chapter 3: 4가지 아키텍처 패턴
- 목표: 파이프라인, 팬아웃/팬인, 전문가풀, 생성-검증 4가지 패턴을 구분하고, 도메인에 맞는 패턴을 선택할 수 있다
- 핵심 내용:
  - 파이프라인: 순차 의존 작업 (A → B → C)
  - 팬아웃/팬인: 병렬 독립 작업 후 결과 통합
  - 전문가풀: 상황별 라우터가 적절한 전문가 선택 호출
  - 생성-검증: Producer → Reviewer → (문제시) 재생성 루프
  - 패턴 선택 의사결정 트리
  - 패턴 조합: 실전에서는 여러 패턴이 Phase별로 혼합됨
- 예제: 각 패턴별 실제 사례 1개씩 (SF 소설, 웹툰, 리서치 팀 등)
- 인포그래픽: 4가지 패턴 흐름도 + 선택 가이드 플로차트
- 분량: 14~16페이지

### Part 2: 실전 — 첫 번째 하네스 만들기

#### Chapter 4: 에이전트 정의 작성하기
- 목표: `.claude/agents/` 에 에이전트 정의 파일을 올바르게 작성할 수 있다
- 핵심 내용:
  - 에이전트 파일의 전체 구조 (frontmatter + 본문)
  - frontmatter 작성법: name, description, model 선택 기준
  - 본문 섹션별 작성법: 핵심 역할, 작업 원칙, 출력 형식, 협업
  - description에 트리거 키워드를 효과적으로 넣는 법
  - model 선택 가이드: sonnet vs opus, 비용 대비 품질 판단
  - 좋은 에이전트 정의 vs 나쁜 에이전트 정의 비교
- 예제: "코드 리뷰어" 에이전트를 step-by-step으로 작성
- 분량: 12~14페이지

#### Chapter 5: 스킬 정의 작성하기
- 목표: `.claude/skills/*/SKILL.md` 에 스킬 파일을 작성하고, references 분리를 할 수 있다
- 핵심 내용:
  - 스킬 파일 구조: frontmatter + 워크플로우/절차 + 도구 사용법 + 출력 규칙
  - 에이전트 1개 : 스킬 N개 관계 설계
  - 여러 에이전트가 공유하는 스킬 설계
  - references/ 디렉토리 활용: 큰 참고자료 분리
  - 워크플로우 작성시 Phase별 검증 기준 명시의 중요성
- 예제: "코드 리뷰" 스킬을 step-by-step으로 작성 (Chapter 4의 에이전트와 연결)
- 분량: 12~14페이지

#### Chapter 6: 오케스트레이터 스킬 만들기
- 목표: 에이전트 팀을 조율하는 오케스트레이터 스킬을 작성할 수 있다
- 핵심 내용:
  - 오케스트레이터의 역할: 에이전트 간 데이터 흐름과 실행 순서 관리
  - 에이전트 구성 테이블 작성법 (에이전트, 역할, 모델, Phase)
  - Phase별 워크플로우 설계: 입력/출력/검증 기준
  - 에이전트 간 산출물 전달 규칙 (파일 기반 통신)
  - 디렉토리 구조 설계: 산출물이 저장될 위치 계획
  - "각 Phase 완료 후 사용자 확인" 패턴의 중요성
- 예제: Chapter 4~5의 에이전트/스킬을 묶어 "코드 품질 관리" 오케스트레이터 완성
- 인포그래픽: 오케스트레이터 실행 흐름도
- 분량: 14~16페이지

### Part 3: 심화 — 실전 하네스 설계 사례

#### Chapter 7: 사례 1 — 책 집필 하네스 (파이프라인 + 생성-검증)
- 목표: 이 책을 만든 book-writer 하네스의 전체 구조를 이해하고, 복합 패턴 설계를 할 수 있다
- 핵심 내용:
  - book-writer 하네스 전체 구조 분석
  - 5명의 에이전트: book-planner, chapter-writer, book-editor, infographic-creator, book-publisher
  - Phase 1~5 워크플로우 상세 분석
  - 파이프라인 패턴 (Phase 1→2→4→5)과 생성-검증 패턴 (Phase 3) 조합
  - 각 에이전트의 정의 파일 해부
  - 산출물 디렉토리 구조와 파일 네이밍 규칙
- 예제: book-writer 하네스를 처음부터 재현하며 구축
- 인포그래픽: book-writer 하네스 전체 아키텍처 다이어그램
- 분량: 16~18페이지

#### Chapter 8: 사례 2 — 리서치 팀 하네스 (팬아웃/팬인)
- 목표: 병렬 작업이 필요한 도메인에서 팬아웃/팬인 패턴을 적용할 수 있다
- 핵심 내용:
  - 리서치 도메인 분석: 웹 검색, 데이터 분석, 보고서 작성
  - 병렬 에이전트 설계: 동시 검색 → 결과 통합 → 보고서 생성
  - 팬아웃 시 작업 분배 전략
  - 팬인 시 결과 통합 전략 (충돌 해결, 우선순위)
  - 병렬과 순차의 적절한 조합
- 예제: 3명의 리서처 + 1명의 분석가 + 1명의 보고서 작성자로 구성된 하네스 구축
- 인포그래픽: 팬아웃/팬인 데이터 흐름도
- 분량: 14~16페이지

#### Chapter 9: 사례 3 — 코드 생성 하네스 (전문가풀 + 생성-검증)
- 목표: 전문가풀 패턴으로 도메인별 전문가를 라우팅하고, 생성-검증 루프를 구현할 수 있다
- 핵심 내용:
  - 다중 언어/프레임워크 코드 생성 시나리오
  - 라우터 역할의 오케스트레이터 설계
  - 전문가별 에이전트 설계: Python, Java, Kotlin 등
  - 생성된 코드의 자동 검증 (테스트 실행, 린트)
  - 검증 실패 시 재생성 루프 설계
  - 전문가풀 확장: 새 전문가 추가 시 기존 구조 변경 최소화
- 예제: 3개 언어 전문가 + 코드 리뷰어로 구성된 하네스 구축
- 분량: 14~16페이지

### Part 4: 운영 — 하네스 관리와 확장

#### Chapter 10: 하네스 메타 스킬 활용하기
- 목표: harness 메타 스킬을 사용하여 새로운 하네스를 자동 생성할 수 있다
- 핵심 내용:
  - harness 메타 스킬의 6단계 워크플로우 (도메인 분석 → 검증)
  - "하네스 구성해줘"로 시작하는 자동 생성 프로세스
  - 메타 스킬이 생성하는 산출물 체크리스트
  - 기존 에이전트/스킬과의 충돌 방지
  - 생성된 하네스의 검증 절차 (6항목 체크리스트)
- 예제: 새 도메인(예: "기술 블로그 자동화")에 대해 harness 스킬로 하네스 자동 생성
- 분량: 12~14페이지

#### Chapter 11: 디버깅과 개선
- 목표: 하네스 운영 중 발생하는 문제를 진단하고 개선할 수 있다
- 핵심 내용:
  - 흔한 문제 패턴: 에이전트 간 산출물 불일치, 무한 루프, 컨텍스트 초과
  - 에이전트 description 트리거 키워드 미스매치 디버깅
  - 오케스트레이터 Phase 실패 시 복구 전략
  - 에이전트 분리/통합 판단: 언제 에이전트를 쪼개고 합칠 것인가
  - 성능 최적화: model 선택(sonnet vs opus), 컨텍스트 절약
  - 반복 실행 결과의 품질 안정화 전략
- 예제: 실패하는 하네스를 단계별로 디버깅하여 수정하는 시나리오
- 분량: 12~14페이지

#### Chapter 12: 나만의 하네스 설계하기
- 목표: 독자 자신의 도메인에 맞는 하네스를 처음부터 설계하고 구축할 수 있다
- 핵심 내용:
  - 도메인 분석 체크리스트: 작업 유형 식별, 전문 영역 분해
  - 아키텍처 패턴 선택 의사결정 트리 (종합)
  - 에이전트 분리 기준 4가지 재점검
  - 점진적 구축 전략: 에이전트 1개부터 시작하여 확장
  - 하네스 설계 워크시트 (빈 템플릿 제공)
  - 최종 체크리스트: 올바른 하네스인지 자가 검증
- 예제: 독자가 직접 채우는 설계 워크시트 (빈칸 채우기 형식)
- 인포그래픽: 하네스 설계 전체 프로세스 요약 플로차트
- 분량: 14~16페이지
